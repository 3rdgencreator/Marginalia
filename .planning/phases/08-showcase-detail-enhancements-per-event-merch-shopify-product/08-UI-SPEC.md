## Spacing Scale

Declared values — all tokens already defined in `app/globals.css @theme {}`:

| Token | CSS Variable | Value | Usage in this phase |
|-------|-------------|-------|---------------------|
| xs | `--space-xs` | 4px | Gap between recording title and dj_label; icon-label gap |
| sm | `--space-sm` | 8px | Margin below section label (LISTEN / MERCH / LINKS) |
| md | `--space-md` | 16px | `mb` below each section heading; gap between link rows |
| lg | `--space-lg` | 24px | `mb` of flyer block, consistent with existing `max-w-2xl` block |
| xl | `--space-xl` | 32px | `mb` of back-link bar; consistent with existing header blocks |
| 2xl | `--space-2xl` | 48px | `mb` of each top-level section block (recordings / merch / links) |
| 3xl | `--space-3xl` | 64px | Container `py` — already in `page.tsx`, do not change |

Exceptions:

- **Between consecutive recordings** (multi-embed): `gap-(--space-xl)` (32px) in a `flex flex-col` wrapper.
  Rationale: each embed is 166px tall; 32px breathing room prevents section from feeling dense.
- **Admin repeater row internal gap**: `gap-(--space-sm)` (8px) between inputs inside one row.
  Rationale: standard spacing token; admin density preserved without breaking the `{4, 8, 16, 24, 32, 48, 64}` scale.

## Typography

All roles use `var(--font-sans)`. No new sizes or weights are introduced.

| Role | CSS Variable | Rendered Size | Weight | Line Height | Usage in this phase |
|------|-------------|--------------|--------|-------------|---------------------|
| Section heading | `--text-heading` | 24px (→ 32px desktop) | 700 | `--leading-tight` (1.1) | "LISTEN", "MERCH", "LINKS" h2 headings |
| Recording title | `--text-body` | 16px | 700 | `--leading-snug` (1.2) | `<h3>` per recording (title field) |
| dj_label / subtitle | `--text-body` | 16px | 400 | `--leading-snug` (1.2) | Appended to title in same `<h3>` with muted color |
| Link label | `--text-body` | 16px | 400 | `--leading-normal` (1.5) | Text label in links list |
| Merch price | `--text-label` | 12px | 400 | `--leading-label` (1.4) | Reused from MerchGrid — no change |
| Section label (eyebrow) | `--text-label` | 12px | 700 | `--leading-label` (1.4) | `uppercase tracking-widest` — matches existing "Past Showcase" pattern |

Typography rules:

- Section headings (`<h2>`) use the same existing class string as the current "Listen" / "Gallery" / "Aftermovie" headings:
  `text-(--text-heading) text-(--color-text-primary) mb-(--space-md) uppercase tracking-widest`
- Recording titles (`<h3>`) do NOT use uppercase. They render as mixed-case title text.
- `dj_label` is rendered inline after the title, separated by a space-padded em-dash: `title — dj_label`.
  The dj_label portion uses `text-(--color-text-muted)` with weight 400.
- Admin form labels use `text-xs text-gray-500 uppercase tracking-widest` (existing admin pattern from `EditShowcasePage`).

## Section Layout Contract (Public Page)

**Primary visual anchor (focal point):** the existing event flyer — it remains at the top of the page and is the dominant visual element. All new sections (Links, Merch, Listen) are secondary; their hierarchy descends in visual weight: Links (lightweight text) → Merch (image grid) → Listen (audio embeds with title text). Header, flyer, and upcoming CTAs precede every new section to preserve the existing focal hierarchy.

Render order in `app/showcases/[slug]/page.tsx` after this phase:

```
1. Back link                    — existing
2. Header (title + date + loc)  — existing
3. Flyer                        — existing
4. Upcoming CTAs                — existing (isPast=false only)
5. [NEW] Links section          — always, when links.length > 0
6. [NEW] Merch section          — always, when merch_handles.length > 0
7. Listen (Recordings)          — isPast=true only, when recordings.length > 0
8. Aftermovie                   — isPast=true only (existing)
9. Gallery                      — isPast=true only (existing)
```

Rationale for Links and Merch placement:
- Links (sponsor, after-party venue, etc.) are relevant for upcoming events — appear before past-only content.
- Merch (event t-shirts) also relevant pre-event for purchasing. Placed after links because links are lightweight text while merch is visually heavier grid.
- Recordings, Aftermovie, Gallery are post-event content — remain grouped at the bottom, gated by `isPast`.

## Admin Form Contract

### Recordings Repeater Section

Placed in `app/admin/(protected)/showcases/[slug]/page.tsx` as a new `<Section>` block, below the existing Links section.

Section title: "Recordings"

Each row in the repeater:
```
[Title* (text, flex-1)] [DJ Label (text, flex-1)] [Sort Order (number, w-20)] [× Remove]
```

Rendered using existing `<Field>` and `<Grid2>` primitives from `@/components/admin/AdminField`.

- `sort_order` is a `type="number"` input with `defaultValue={0}`. Width: `w-20` (narrow). No drag handle.
  Rationale: sort_order number input is simpler to implement than drag-and-drop, keeps admin consistent with existing photo order pattern, and recording count is low (1–5). Claude's discretion resolved: use sort_order number.
- "Remove" is a small `<button type="submit">` in a separate `<form action={deleteRecording.bind(null, recording.id)}>`, styled:
  `px-2 py-1 text-xs text-(--color-destructive) border border-(--color-destructive)/30 hover:bg-(--color-destructive)/10 transition-colors duration-150`
- "Add Recording" button at bottom of section:
  `px-4 py-2 bg-white/10 text-white text-sm font-bold uppercase tracking-widest hover:bg-[#9EFF0A] hover:text-black transition-colors duration-150`
  (Matches existing "+ Add Photo" button in the same admin page)

The "Add Recording" action creates a new recording row via server action `addShowcaseRecording(showcaseId)` with empty defaults. The admin reloads to show the new row.

### Links Repeater Section

Section title: "Links"

Each row:
```
[Label (text, flex-1)] [URL (url, flex-1)] [× Remove]
```

Same remove button pattern as Recordings. Same "Add Link" button pattern.

No `sort_order` for links — order is determined by jsonb array index. Admin can re-order by removing and re-adding (acceptable for low-volume data; v2 can add sort if needed).

### Merch Multi-Select Section

Section title: "Event Merch"

UI pattern: Async-loaded checkbox list.

```
[Loading state] → checkbox list of Shopify products
```

Structure:
- `"use client"` component: `components/admin/ShowcaseMerchPicker.tsx`
- On mount: `fetch('/api/admin/shopify-products')` → returns `{ handle, title, image }[]`
  (New API route, or reuse existing Shopify fetch; the route calls `fetchShopifyProducts` with no handle filter)
- Loading state: `<p className="text-xs text-gray-500">Loading products…</p>`
- Error state: `<p className="text-xs text-red-400">Could not load Shopify products. Check SHOPIFY_* env vars.</p>`
- Each product rendered as:
  ```html
  <label className="flex items-center gap-3 cursor-pointer hover:text-white transition-colors duration-150">
    <input
      type="checkbox"
      name="merch_handles"
      value={product.handle}
      defaultChecked={currentHandles.includes(product.handle)}
      className="accent-[#9EFF0A] w-4 h-4"
    />
    <span className="text-sm text-(--color-text-secondary)">{product.title}</span>
    <span className="text-xs text-(--color-text-muted) ml-auto">{product.handle}</span>
  </label>
  ```
- Product count expected: < 50. No typeahead needed. Simple scrollable list in a `max-h-64 overflow-y-auto` container.
- `accent-[#9EFF0A]` on checkboxes for brand consistency.

The checked handles are submitted as `merch_handles[]` array via the existing `updateShowcase` server action (which reads `formData.getAll('merch_handles')`).

Empty states (public page):
- Sections are not rendered when empty (D-11/D-12 in CONTEXT.md). No empty state copy needed on the public page.

Error states:
- Invalid SoundCloud URL in recording: not validated client-side. SoundCloudEmbed renders a broken iframe. The admin label hint reads: `SoundCloud track or set URL` to guide correct input.
- Shopify API unavailable (merch picker): error message shown inline (see above).

Destructive actions in this phase:
- **Remove recording row**: No confirmation dialog. The row count is visible; accidental removal is recoverable by re-adding. Matches existing "Delete Photo" pattern which also has no confirmation.
- **Remove link row**: Same — no confirmation.
- **Delete showcase** (existing action, not changed): existing `<DeleteButton>` with existing confirmation — no change.
